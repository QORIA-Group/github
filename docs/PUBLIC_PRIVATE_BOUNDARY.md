# Public / Private Boundary

**QORWAY Technology — Public Architecture Repository**

> Public documentation builds trust. Private repositories preserve implementation depth, proprietary architecture, and tenant safety.

---

## 1. Purpose

This document defines the boundary between the public QORWAY architecture repository and the private QORWAY core repositories.

The public repository explains the architecture in a public-safe form.

The private core contains implementation-preparation material that makes the architecture executable.

---

## 2. Public Repository Scope

The public repository may contain:

- architectural principles
- layer responsibilities
- public-safe system maps
- high-level Domain Pack concepts
- public examples
- public interface standards
- governance principles
- security and sovereignty doctrine
- non-sensitive decision trace examples
- repository and website narrative alignment material

The public repository should help a technical, institutional, or enterprise reader understand what QORWAY is without exposing private implementation depth.

---

## 3. Private Repository Scope

Private repositories contain internal architecture material such as:

- implementation-preparation specifications
- private Domain Pack details
- internal registries
- graph projection details
- causal rule libraries
- PolicyCore validation details
- PulseFlow orchestration contracts
- GreenCore routing logic
- Atlas internal alignment material
- controlled prompt material
- internal scoring models
- tenant-specific extension logic
- sensitive IP and research material

These materials are not part of the public trust layer.

---

## 4. Public-Safe Transformation Rule

Private material may become public only after it is transformed into:

- concept-level explanation
- non-sensitive architecture doctrine
- implementation-neutral terminology
- no proprietary thresholds
- no operational rule tables
- no tenant material
- no controlled prompt content
- no private registries

Public material must describe responsibilities, not expose execution internals.

---

## 5. Domain Pack Boundary

Public documentation may describe Domain Packs as the Context & Business Intelligence Layer of QORWAY.

It may describe examples such as Finance, Supply Chain, CSRD/RSE, Regulator, Workforce Capital, and Sovereign Resilience at a high level.

It must not publish private Domain Pack internals, private scoring logic, tenant extensions, or implementation contracts.

---

## 6. Website Boundary

The public website must derive its technical narrative from this public repository, not directly from private repositories.

The website may simplify language for executives, institutions, or partners.

It must not introduce claims that are not supported by the public architecture repository.

---

## 7. ODC Boundary

ODC may act as a dogfooding cockpit, service layer, and operating evidence source.

ODC must not contain QORWAY runtime internals.

QORWAY public documentation must not imply that ODC is the QORWAY runtime.

---

## 8. Final Rule

```text
Private Core defines the depth.
Public Repository defines the trust layer.
Website communicates the public narrative.
ODC provides operating evidence.
```
